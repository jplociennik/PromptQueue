# Frontend (React SPA — od pq-4) — szczegóły

- Stack: Vite 6 + React 18.3 + TS strict (szablon react-ts). Testy: Vitest 3 + RTL 16, `globals: false`, `setupTests.ts`=`@testing-library/jest-dom/vitest`. Build=`tsc -b && vite build`; testy w `tsconfig.app` (include `src`), `noEmit`→nie trafiają do dist.
- STRUKTURA = skill code-frontend REGUŁA 9 (EGZEKWUJ; superseduje starszy szkic pq-4.md z `ui/` małą literą): folder-per-component PascalCase + współlokowany `Nazwa.module.css`, BEZ barreli `index`, import wprost pliku. Reużywalne→`components/UI/` (UI wielką). Domenowe→`features/<f>/<Comp>/`. WSZYSTKIE custom hooki płasko w `hooks/` (też feature'owe, np. `usePromptFields`). Typy współdzielone + klient API = pliki (`api/*.ts`, `features/<f>/types.ts`).
- HTTP w pq-4 = **axios** (`http` instance + `ApiError` + czysta `toApiError` + interceptor `Promise.reject(toApiError(e))`) — ODSTĘPSTWO od doc (mówił `fetch`+`apiFetch`); poprawne, doc do zsynchronizowania. `toApiError`: `status=response?.status??0` (sieć→0), `message=data?.title??error.message`, `validationErrors=data?.errors`. Testowana jako pure fn bez sieci (fake przez `Parameters<typeof toApiError>[0]`+`as unknown as`, bez `any`). `createPrompts` zwraca `data: CreatePromptsResponse`, ma `signal?` (dla pq-5).
- `api/types.ts` = wierne lustro pq-2 (`PromptContracts.cs`): `PromptStatus` camelCase, `CreatePromptsResponse {ids:string[]; status}`, `PromptResponse` (result/errorMessage nullable, createdAt/updatedAt) pre-deklarowany dla pq-5, `ValidationErrors=Record<string,string[]>`. Backend 400=`Results.ValidationProblem` (title domyślny EN "One or more validation errors occurred.", detale w `errors`); 500 title EN "An unexpected error occurred." — front pokazuje surowo.
- WZORCE (potwierdzone, NIE flaguj): reset formularza TYLKO po sukcesie (`submit:Promise<boolean>`, `reset()` gdy `ok`); blokada double-submit `disabled` gdy `state.status==='submitting'` (Wyślij/Dodaj/Usuń/pola); reducery `useReducer` używają `switch` (wyjęte spod "słownik-nie-switch"); lazy init `useReducer(r, undefined, ()=>[createField()])` bo `crypto.randomUUID`; BRAK przedwczesnej memoizacji (mała lista→OK bez useCallback); a11y `aria-label="Prompt N"`+`aria-invalid`+`role="alert"`; CSS Modules; TextArea chat-input pure-CSS (`field-sizing: content`+min/max-height fallback).
- POWTARZALNE ZNALEZISKA (frontend): (1) usunięcie ostatniego pola→`fields=[]`; `[].some()===false` omija guard pustości→pusty POST→400 (fix: min 1 pole / blok gdy `length===0`). (2) `validationErrors` łapane+testowane, ale NIE renderowane — Alert pokazuje tylko `title` (generyczny EN)→sprawdzaj czy detale 400 docierają do UI. (3) `globals:false`→RTL auto-cleanup WYŁĄCZONY; przy testach renderujących (pq-5) dodać `afterEach(cleanup)` lub `globals:true`.
- HANDOFF: pq-5 `getPrompts`+polling (useEffect cleanup, `signal` już w `createPrompts`); pq-6 `crypto.randomUUID` wymaga secure context (localhost/https).
